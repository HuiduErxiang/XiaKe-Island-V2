## 原则 1: Git 分支管理

### 1.1 分支定义

| 分支 | 用途 | 保护状态 |
|------|------|----------|
| `master` | 唯一可部署生产环境的分支 | 🔒 保护分支，仅 PR 合并 |
| `feature/*` | 需求开发分支 | - |
| `fix/*` | 修复开发分支 | - |

### 1.2 禁止行为
- ❌ **严禁直接推送代码到 master 分支**
- ❌ **严禁在 master 分支直接修改代码**

## 原则 2: 开发规范流程（四步法）

所有代码改动必须遵循以下四步流程：

### Step 1: 创建需求分支

根据当前 Git 状态，按以下规则处理：

```
if 当前分支 == master:
    → 新建分支: git checkout -b feature/xxx-需求名称
    
elif 当前分支 != master:
    if 工作区不干净 (有未提交改动):
        → 继续在当前分支开发
        
    else (工作区干净):
        → 重新从 master 拉新分支
        git checkout master && git pull origin master
        git checkout -b feature/xxx-新需求名称
```

**检查工作区状态命令**: `git status --porcelain`

### Step 2: 开发环境部署验收

- 在本地需求分支完成代码改动
- **不推送到远程**
- 直接部署到**开发环境** (`/dev/`)
- 用户验收开发环境功能

### Step 3: 用户指令后提交远程

> **关键**: 必须等待用户明确指令 `"提交代码"` 或 `"push 代码"`

收到指令后：
```bash
git add -A
git commit -m "feat: xxx"
git push origin feature/xxx-需求名称
```

然后在 GitHub 创建 PR → master

### Step 4: 合并后部署生产

> **关键**: 必须等待用户在远端确认 `"已合并"` 或 `"合并完成"`

收到确认后：
```bash
git checkout master
git pull origin master
./deploy.sh
```

## 原则 3: 环境定义

| 环境 | 地址 | 用途 | 部署来源 |
|------|------|------|----------|
| 开发环境 | `/dev/` | 功能验收 | 任意本地分支 |
| 生产环境 | `/` | 线上服务 | 仅 `master` 分支 |

---

## 原则 4: AI 助手行为约束

作为 AI 助手，我必须：

1. **绝不自动 push 代码** - 必须等待用户明确指令
2. **绝不自动部署生产** - 必须等待用户确认合并完成
3. **严格区分开发/生产** - 验收前只操作开发环境
4. **保持工作区干净** - 提交前确认改动完整
