# Feature Specification: 藏经阁标签检索 Skill

**Feature Branch**: `001-cang-tag-search-skill`  
**Created**: 2026-04-22  
**Updated**: 2026-04-24  
**Status**: Draft  
**Input**: User description: "让藏经阁标签检索成为产品 Agent 的一个 Skill：产品 Agent 在分析客户材料生成 PRD 的过程中，需要能够从藏经阁知识库中根据标签检索相关文章/知识/证据，并将检索到的结构化证据注入 PRD 草稿中。该 Skill 应支持：1) 获取藏经阁所有可用标签；2) 根据标签检索匹配的知识资产（PDF 文献）；3) 从检索结果中按相关性排序并提取结构化证据片段。"

> **📌 数据来源变更说明**：cang 仓库（`huidu/cang`）已停止作为活跃数据源，仅保留 Git 历史用于追溯。当前藏经阁的物理数据存储在素材平台 `uploads/` 目录，元数据（分类、标签、文件映射）由素材平台 SQLite 数据库（`database.sqlite`）统一管理。本 Skill 的数据访问方式从"扫描 cang 文件系统"变更为"查询素材平台数据库 + 读取 uploads 目录"。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 产品 Agent 根据客户材料中的产品名检索相关证据 (Priority: P1)

产品 Agent 收到客户关于"仑卡奈单抗"的材料后，需要自动从藏经阁检索该产品的相关 PDF 文献证据（临床试验数据、疗效数据、安全性数据）以及相关的疾病知识，将结构化证据注入 PRD 草稿的"证据支撑"章节。

**Why this priority**: 这是数据驱动内容生产的核心链路——没有证据注入，PRD 就只能依赖 AI 幻觉，失去藏经阁作为"知识真相源"的价值。该能力直接决定产品 Agent 产出物的专业可信度。

**Independent Test**: 可以独立测试：给定关键词"lecanemab"或"仑卡奈单抗"，Skill 能查询素材平台数据库返回所有匹配的知识资产（关联标签的 PDF 文献），并按分类分组展示。

**Acceptance Scenarios**:

1. **Given** 产品 Agent 正在处理包含"仑卡奈单抗"的客户材料，**When** 调用标签检索 Skill 传入关键词"仑卡奈单抗"，**Then** Skill 查询素材平台 `tags` + `material_tags` + `cang_materials` 表，返回匹配的知识资产列表，包含相关 PDF 文献的引用和分类信息
2. **Given** 产品 Agent 需要获取某个疾病领域的全景知识，**When** 调用标签检索 Skill 传入"阿尔茨海默病"，**Then** Skill 返回关联的 PDF 文献，按分类（如肿瘤、神经、指南共识等）和相关性排序

---

### User Story 2 - 产品 Agent 获取藏经阁完整的标签体系用于检索意图生成 (Priority: P2)

产品 Agent 在分析客户材料后，需要了解藏经阁中都有哪些可用的标签/分类，以便生成更精准的检索意图。例如：知道藏经阁有"肿瘤免疫"、"ADC药物"、"PD-1"等标签，可以从材料中提取这些关键词进行精确检索。

**Why this priority**: 检索效果取决于检索意图的质量。如果产品 Agent 不知道藏经阁里有什么标签，就只能做模糊匹配，召回率和精确率都会下降。获取标签体系是优化检索效果的前置步骤。

**Independent Test**: 可以独立测试：调用 `get_all_tags()` 接口，返回的标签列表应覆盖素材平台 `tags` 表中的所有标签（产品名、疾病名、分类、会议等维度）。

**Acceptance Scenarios**:

1. **Given** 产品 Agent 准备生成检索意图，**When** 调用获取所有标签接口，**Then** 返回素材平台 `tags` 表中当前所有可用的标签，以及 `categories` 表中的分类层级，不含重复项
2. **Given** 返回的标签列表，**When** 产品 Agent 从中选择与当前任务相关的标签，**Then** 可以用这些标签进行精确的按标签检索，返回的结果相关性高于纯关键词匹配

---

### User Story 3 - 从检索结果中提取结构化证据片段注入 PRD (Priority: P3)

产品 Agent 在获得检索结果后，不需要读取完整的 PDF 文件，而是需要 Skill 从中提取出与当前 PRD 主题最相关的结构化证据片段（如：临床试验的关键数据、适应症的描述、安全性的要点），直接注入 PRD 草稿。

**Why this priority**: 产品 Agent 不应直接处理文件的原始内容，而应通过 Skill 获取已提取和格式化的证据。这保持了 Skill 的封装性，也方便后续对证据提取逻辑进行独立迭代优化。

**Independent Test**: 可以独立测试：给定一个知识资产的数据库记录（包含 `url` 指向 uploads 目录下的 PDF），调用证据提取接口，返回的是按主题分组的结构化证据片段，而非原始 PDF 的全量内容。

**Acceptance Scenarios**:

1. **Given** 标签检索返回了某份 PDF 文献的数据库记录，**When** 调用证据提取传入该记录和主题"疗效数据"，**Then** 从 uploads 目录下对应的 PDF 中提取结构化的疗效证据片段（临床试验名称、主要终点、关键数据）
2. **Given** 标签检索返回了多个知识资产，**When** 对每个资产调用证据提取，**Then** 返回的证据片段包含来源追溯信息（数据库记录 ID、PDF 文件名、分类、更新时间），便于 PRD 中引用

---

### Edge Cases

- **What happens when 检索关键词在藏经阁中没有匹配？** 返回空列表并附带提示信息，不抛出异常，不影响产品 Agent 的后续流程
- **What happens when 素材平台数据库结构发生变化？** Skill 应通过 SQL 查询动态适配，不依赖硬编码的表结构；表结构变更时需在配置中更新字段映射
- **How does system handle 同一标签匹配到大量资产？** 返回结果应分页或限制数量（默认最多 50 条），并按分类相关性和标签匹配度排序
- **How does system handle 证据提取时 PDF 文件不存在或格式异常？** 返回空证据列表并记录错误信息，不中断整体检索流程

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Skill 必须提供获取藏经阁所有可用标签的接口，标签来源为素材平台数据库的 `tags` 表和 `categories` 表，覆盖产品名、疾病名、分类、会议等多维度
- **FR-002**: Skill 必须提供根据单个或多个标签检索知识资产的接口，通过查询 `material_tags` + `cang_materials` 表实现；传入多个标签时按 OR 逻辑检索，返回结果按匹配标签数量降序排列（匹配越多排序越靠前），并标注每个资产匹配的标签数量和匹配度
- **FR-003**: Skill 必须支持从检索到的知识资产中提取结构化证据片段，证据片段应包含主题、内容、来源追溯信息
- **FR-004**: Skill 的标签索引直接从素材平台数据库的 `tags` 表和 `categories` 表读取，不依赖外部手动维护的标签列表；cang 仓库的原始目录结构仅作为历史追溯，不参与当前标签生成
- **FR-005**: 检索结果必须按匹配标签数量降序排列（匹配越多排序越靠前），并标注每个资产匹配的标签数量和匹配度。注意：原始 FR-005 定义"分类相关性优先、标签匹配度第二"的排序策略已延期——当前实现按 `match_count DESC, updated_at DESC` 排序，语义相似度排序待未来迭代。
- **FR-006**: 当检索无结果时，Skill 必须返回空列表和友好的提示，不得抛出异常或中断调用方流程
- **FR-007**: 证据提取必须支持按主题过滤（如"疗效数据"、"安全性数据"、"适应症"），只返回与主题相关的片段
- **FR-008**: Skill 每次调用必须记录操作日志，包含调用时间、输入标签、检索返回的资产数量、提取的证据片段来源路径、执行耗时等信息，日志按日期分文件存储于项目 `logs/` 目录

### Key Entities

- **KnowledgeAsset**: 藏经阁中的知识资产，代表一个可被检索和消费的知识单元。属性包括：数据库 ID（`cang_materials.id`）、资产名称（`name`）、分类（`category_id` 关联 `categories` 表）、存储路径（`url` 指向 uploads 目录下的 PDF）、关联标签（通过 `material_tags` 关联）、文件 Hash（`file_hash`）、最后更新时间（`updated_at`）
- **Tag**: 用于检索知识资产的关键词或分类标签。属性包括：标签名（来自 `tags.name`）、标签类型（来自 `categories.name`）、关联资产数量（通过 `material_tags` 统计）
- **EvidenceFragment**: 从知识资产中提取的结构化证据片段。属性包括：片段内容、所属主题、来源资产的数据库 ID、来源 PDF 文件名、来源分类、提取时间、相关性分数
- **SearchResult**: 标签检索的返回结果。属性包括：匹配资产列表、总匹配数、使用的标签、排序方式、分页信息

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 给定藏经阁中已存在的任意产品名（如"lecanemab"、"T-DXd"）或疾病名（如"阿尔茨海默病"、"胃癌"），标签检索返回所有匹配的知识资产，召回率 ≥ 95%（即素材平台数据库中所有包含该标签的资产都被检索到）
- **SC-002**: 获取所有标签接口返回的标签数量 ≥ 素材平台 `tags` 表中的独立标签总数，准确率 100%（无重复标签、无遗漏）
- **SC-003**: 证据提取返回的结构化片段包含完整的来源追溯信息（数据库记录 ID、PDF 文件名、分类、更新时间），产品 Agent 可以将这些追溯信息直接写入 PRD 的引用部分
- **SC-004**: 当检索关键词在藏经阁中无匹配时，Skill 在 100ms 内返回空结果和提示信息，调用方流程不受任何影响

## Clarifications

### Session 2026-04-22 (Original)

- **Q1**: Skill 的调用时机和方式 → **A**: 作为产品 Agent Speckit 流程中"证据补充"步骤的固定环节统一调用，Skill 只接收标签/分类等固定关键词，不暴露复杂检索逻辑给产品 Agent
- **Q3**: 多标签检索的逻辑关系 → **C**: 传入多个标签时按 OR 逻辑检索，返回结果按匹配标签数量降序排列（匹配越多排序越靠前），并标注每个资产匹配的标签数量和匹配度
- **Q4**: 检索执行的日志与审计 → **A**: 每次 Skill 调用都写入 JSON 日志文件，记录时间、输入标签、返回资产数量、提取的证据来源路径等信息，存入 `XiaKe-Island-V2/logs/` 目录下，按日期分文件存储

### Session 2026-04-23 (Update)

- **Q5**: 数据来源从 cang 文件系统变更为素材平台数据库，是否需要修改接口契约？ → **A**: 接口契约保持不变（输入输出格式不变），仅内部实现从"文件系统扫描"替换为"SQLite 数据库查询 + uploads 目录读取"
- **Q6**: cang 仓库中的历史数据（如 L1-L4 目录结构、product_manifest.json）如何处理？ → **A**: cang 仓库仅保留 Git 历史用于追溯，不再作为活跃数据源；如需访问历史结构化数据，可通过 Git 回溯特定提交
- **Q7**: 素材平台数据库与 XiaKe-Island-V2 的部署关系？ → **A**: 两者部署在同一台服务器上，Skill 通过本地文件路径直接访问素材平台 SQLite 数据库和 uploads 目录，不需要网络 API 调用

## Assumptions

- 素材平台数据库（`ai-awesome-material-platform/backend/database.sqlite`）与 XiaKe-Island-V2 部署在同一台服务器上，Skill 通过 SQLite 本地连接直接读取元数据，通过文件系统直接读取 uploads 目录下的 PDF 文件
- 素材平台数据库的数据结构在实现期间保持稳定（`cang_materials`、`tags`、`material_tags`、`categories` 表结构等）
- 标签检索的"相关性排序"当前实现按匹配标签数量降序排列（`match_count DESC, updated_at DESC`），原始设计中"分类匹配度优先、标签匹配度第二"的排序策略已延期，待后续迭代实现语义相似度排序
- 证据提取的初始版本支持 PDF 格式的知识资产，通过读取 uploads 目录下的 PDF 文件进行内容提取
- 藏经阁当前已有 309 份唯一 PDF（uploads 目录 315 个文件含 6 组重复），标签索引通过查询素材平台数据库实时获取，不维护任何持久化或内存缓存
- cang 仓库（`huidu/cang`）不再作为活跃数据源，仅用于 Git 历史追溯
